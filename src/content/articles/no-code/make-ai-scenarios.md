Make（旧Integromat）にAI APIを組み込むと、問い合わせ対応・分析・社内業務を自動化できます。
この記事では、Make AIテンプレートとして使える実務向けシナリオ5選を、モジュール設定とプロンプト付きで紹介します。
コードを書かないGUI（画面上のマウス操作で設定をおこなう方式）だけで、AI業務自動化を構築できます。

## MakeでAI APIを呼び出す基本

MakeでAI APIを呼び出すとは、HTTPモジュールまたはOpenAI公式モジュールを使ってAIにテキスト処理をリクエストする操作のことです。

### 接続方法の比較

| 方法 | 対応API | 設定難易度 | 使い分け |
|------|--------|----------|---------|
| OpenAI公式モジュール | OpenAI（ChatGPT） | 簡単（APIキーのみ） | OpenAIだけ使う場合 |
| HTTPモジュール | Claude、Gemini、任意のAPI | やや手間（ヘッダー設定が必要） | 複数のAIを切り替える場合 |

### OpenAI公式モジュールの設定

1. シナリオに「OpenAI - Create a Chat Completion」モジュールを追加します
2. ConnectionでOpenAI APIキーを登録します
3. 以下のパラメータを設定します

```json
{
  "model": "gpt-4o-mini",
  "temperature": 0.3,
  "max_tokens": 1024
}
```

- **model**: 用途に応じて選択します。分類や分析には安価な `gpt-4o-mini`、複雑な文章生成には `gpt-4o` が適しています
- **temperature**: 0に近いほど安定した出力になります。分類タスクは `0`〜`0.3`、文章生成は `0.5`〜`0.8` が目安です
- **max_tokens**（AIの応答の最大トークン数を制限するパラメータ）: 短い分類なら `256`、文章生成なら `1024` 以上を設定します

### HTTPモジュールでClaude APIを呼び出す設定

OpenAI以外のAI（Claude、Geminiなど）を使う場合は、HTTP「Make a request」モジュールで直接APIを呼び出します。以下はClaude APIの設定例です。

**URL:**

```text
https://api.anthropic.com/v1/messages
```

**Headers:**

```json
{
  "x-api-key": "{{anthropic_api_key}}",
  "anthropic-version": "2023-06-01",
  "content-type": "application/json"
}
```

**Body（JSON）:**

```json
{
  "model": "claude-sonnet-4-20250514",
  "max_tokens": 1024,
  "messages": [
    {
      "role": "user",
      "content": "以下の問い合わせを分類してください。\n\n{{入力テキスト}}"
    }
  ]
}
```

Makeでは `{{変数名}}` の部分に前のモジュールから渡されたデータが入ります。ConnectionのAPIキーには、各AIサービスの管理画面で発行したキーを登録してください。

## 問い合わせ対応・分析の自動化シナリオ

問い合わせ対応・分析の自動化とは、AIが受信したテキストデータを読み取り、分類や感情判定をおこなって後続の処理に振り分ける仕組みのことです。ここではMake自動化の実例として、2つのシナリオを紹介します。

### シナリオ1: 問い合わせ自動分類

問い合わせ自動分類とは、AIが受信した問い合わせ内容を読み取り、カテゴリを判定して担当者に自動振り分けする処理のことです。

**シナリオ構成:**

```text
[Googleフォーム] → [OpenAI（分類）] → [Router] → [Gmail（通知）]
```

| モジュール | 設定内容 |
|----------|---------|
| Google Forms - Watch Responses | フォームIDを指定 |
| OpenAI - Create a Chat Completion | Model: gpt-4o-mini / System: 分類プロンプト |
| Router（条件分岐で処理の送り先を切り替えるモジュール） | AIの分類結果で通知先を振り分け |
| Gmail - Send an Email | カテゴリ別の担当者に通知 |

**分類プロンプト（System Message に設定）:**

```text
あなたは問い合わせ分類アシスタントです。
問い合わせを以下の4カテゴリに分類し、必ずJSON形式のみで返してください。

カテゴリ: 見積依頼 / 製品問い合わせ / クレーム / その他

出力形式:
{"category":"カテゴリ名","summary":"30文字以内の要約","urgency":"高/中/低"}

分類基準:
- 見積依頼: 金額・費用・見積りに関する内容
- 製品問い合わせ: 機能・仕様・使い方に関する質問
- クレーム: 不具合・不満・返品に関する内容
- その他: 上記に該当しないもの
```

**期待されるレスポンス例:**

```json
{"category":"見積依頼","summary":"法人向け年間契約の見積り依頼","urgency":"高"}
```

Routerの条件設定では、OpenAIモジュールの出力から `category` の値を取得し、値ごとに送信先のメールアドレスを切り替えます。

### シナリオ2: 顧客レビュー感情分析

顧客レビュー感情分析とは、AIが顧客のレビューテキストを読み取り、ポジティブ・ネガティブ・中立を判定してスプレッドシートに記録する処理のことです。

**シナリオ構成:**

```text
[Google Sheets（新規行）] → [OpenAI（感情分析）] → [Google Sheets（結果書き込み）]
```

| モジュール | 設定内容 |
|----------|---------|
| Google Sheets - Watch Rows | レビューデータシートを監視 |
| OpenAI - Create a Chat Completion | レビューの感情をポジティブ/ネガティブ/中立に分類 |
| Google Sheets - Update a Row | 分析結果を同じ行に書き込み |

**感情分析プロンプト（System Message に設定）:**

```text
あなたは顧客レビュー分析アシスタントです。
レビュー文を分析し、必ずJSON形式のみで返してください。

出力形式:
{"sentiment":"positive/negative/neutral","score":0.0-1.0,"keywords":["キーワード1","キーワード2"]}

判定基準:
- positive: 満足・推薦・感謝の表現を含む
- negative: 不満・苦情・改善要望を含む
- neutral: 事実の記述のみで感情表現が少ない
- scoreは確信度（1.0が最も確信が高い）
```

**期待されるレスポンス例:**

```json
{"sentiment":"positive","score":0.85,"keywords":["使いやすい","対応が早い"]}
```

## 社内業務の効率化シナリオ

社内業務の効率化シナリオとは、議事録の要約・メール返信の下書き・SNS投稿文の生成など、日常的に発生するテキスト作成業務をAIに任せる仕組みのことです。

### シナリオ3: 議事録の自動要約

議事録の自動要約とは、Google Docsに保存された会議メモをAIが読み取り、要点と決定事項を抽出してSlackに投稿する処理のことです。

**シナリオ構成:**

```text
[Google Drive（新規ファイル）] → [Google Docs（読み取り）] → [OpenAI（要約）] → [Slack（通知）]
```

| モジュール | 設定内容 |
|----------|---------|
| Google Drive - Watch Files | 議事録フォルダを監視 |
| Google Docs - Get Content | ドキュメントの本文を取得 |
| OpenAI - Create a Chat Completion | 300文字以内で要約 + 決定事項を箇条書き |
| Slack - Create a Message | 要約結果をチャンネルに投稿 |

**要約プロンプト（System Message に設定）:**

```text
あなたは議事録要約アシスタントです。
以下のルールで会議メモを要約してください。

出力形式:
## 会議要約
（300文字以内で要点を記述）

## 決定事項
- （箇条書きで列挙）

## 次回までのTODO
- （担当者名）: （タスク内容）

注意:
- 固有名詞や数値は正確に記載すること
- 決定事項がない場合は「なし」と記載すること
```

### シナリオ4: メール返信ドラフト自動生成

メール返信ドラフト自動生成とは、特定ラベルの受信メールに対してAIが返信案を作成し、下書きとして保存する処理のことです。自動送信ではなく下書き保存にすることで、担当者が内容を確認してから送信できます。

**シナリオ構成:**

```text
[Gmail（新着メール）] → [Filter] → [OpenAI（返信生成）] → [Gmail（下書き保存）]
```

| モジュール | 設定内容 |
|----------|---------|
| Gmail - Watch Emails | 特定ラベルのメールを監視 |
| Filter（条件に一致するデータのみ通過させるモジュール） | 件名に「問い合わせ」を含むメールのみ通過 |
| OpenAI - Create a Chat Completion | メール本文から返信ドラフトを生成 |
| Gmail - Create a Draft | 下書きとして保存（送信は手動確認） |

**返信生成プロンプト（System Message に設定）:**

```text
あなたはビジネスメールの返信ドラフトを作成するアシスタントです。
以下のルールに従って、受信メールへの返信案を作成してください。

ルール:
- 敬語（です・ます調）で統一
- 冒頭に「お問い合わせいただきありがとうございます。」を含める
- 具体的な回答が必要な箇所は【要確認: ○○】と記載し、担当者が確認できるようにする
- 末尾に署名欄のプレースホルダーを入れる
- 200〜400文字程度で簡潔に
```

### シナリオ5: SNS投稿文の自動生成

SNS投稿文の自動生成とは、スプレッドシートに登録した投稿ネタをAIが読み取り、SNS向けの文面を複数パターン生成する処理のことです。

**シナリオ構成:**

```text
[Schedule] → [Google Sheets（ネタ取得）] → [OpenAI（投稿文生成）] → [Google Sheets（結果記録）]
```

| モジュール | 設定内容 |
|----------|---------|
| Schedule | 毎週月曜に実行 |
| Google Sheets - Search Rows | 未生成のネタ行を取得 |
| OpenAI - Create a Chat Completion | ネタから140文字のSNS投稿文を3パターン生成 |
| Google Sheets - Update a Row | 生成結果を記録 |

**投稿文生成プロンプト（System Message に設定）:**

```text
あなたはSNSマーケティングのアシスタントです。
以下のネタから、X（旧Twitter）向けの投稿文を3パターン作成してください。

ルール:
- 各パターンは140文字以内
- ハッシュタグを2〜3個含める
- パターンごとにトーンを変える（1: 情報提供型、2: 共感型、3: 質問型）
- 絵文字は1投稿あたり1〜2個まで

出力形式:
パターン1: （投稿文）
パターン2: （投稿文）
パターン3: （投稿文）
```

## トラブルシューティングと応用

トラブルシューティングと応用とは、AI連携シナリオで発生しやすいエラーへの対処法と、基本シナリオを発展させるカスタマイズ手法のことです。

### よくあるエラーと解決策

| エラー | 原因 | 解決策 |
|--------|------|--------|
| OpenAIモジュールで401エラー | APIキーの設定ミス | ConnectionのAPIキーを再設定 |
| レスポンスが空 | max_tokensが小さすぎる | 512以上に設定 |
| 日本語が文字化け | エンコーディング設定 | HTTPモジュールのParse responseをJSON指定 |
| クレジット上限超過 | Make無料枠（月1,000クレジット）の消費 | シナリオの実行頻度を調整 or [有料プランに移行](https://www.make.com/en/pricing)（2026年2月時点） |
| JSONパースエラー | AIの出力が不安定 | プロンプトに「必ずJSON形式のみで返してください」を追加 |

### 応用・カスタマイズ例

- **複数AIの使い分け** --- 分類はgpt-4o-mini（安価）、要約はClaude Sonnet（高精度）とHTTPモジュールで切り替えることで、コストと品質を両立できます
- **n8nとの併用** --- 大量処理はコスト優位な[n8n × AI連携](/articles/no-code/n8n-ai-workflow)、簡易な連携はMakeと使い分けると運用コストを抑えられます
- **GAS連携** --- MakeのWebhookでGAS（Google Apps Script）を呼び出し、Googleサービスの細かい操作をGASに任せる構成も実用的です

## まとめ

この記事では、Make × AI連携で業務自動化を実現する5つのシナリオをテンプレート付きで紹介しました。

| # | シナリオ | ポイント | 最初に試す価値 |
|---|---------|---------|-------------|
| 1 | 問い合わせ自動分類 | JSON出力で後続処理に連携しやすい | 高い（効果を実感しやすい） |
| 2 | 顧客レビュー感情分析 | スプレッドシート完結で導入が簡単 | 中程度 |
| 3 | 議事録の自動要約 | Slack通知で共有まで自動化 | 高い（会議が多い企業向け） |
| 4 | メール返信ドラフト | 下書き保存で安全に運用できる | 高い（問い合わせ対応が多い企業向け） |
| 5 | SNS投稿文の自動生成 | 複数パターン生成で選択肢を確保 | 中程度 |

最初に試すなら、シナリオ1の問い合わせ自動分類が導入しやすいです。理由は2つあります。プロンプトとモジュール設定をコピーすれば15分程度で構築できること、そしてJSON出力による分類精度が体感で90%以上あり効果を実感しやすいことです。慣れてきたら、シナリオ3の議事録要約やシナリオ4のメール返信ドラフトに展開してみてください。

ノーコードツールの全体像は[ノーコード自動化ツール比較](/articles/no-code/no-code-overview)、AI APIの基礎は[OpenAI API入門ガイド](/articles/ai-api/openai-api-intro)で詳しく解説しています。
