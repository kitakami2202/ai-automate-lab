プロンプト設計とは、AIに意図どおりの出力をさせるために入力文を構造的に組み立てる技術です。
業務自動化でAIを活用する際、プロンプトの書き方次第で出力品質が大きく変わります。
この記事では、4ステップのフレームワークと業務シーン別のテンプレートを使って、AIへの指示の書き方を解説します。

## プロンプト設計フレームワークの全体像

プロンプト設計フレームワークとは、AIへの指示を「役割」「タスク」「出力」「制約」の4要素に分解して構造化する方法です。この4つを漏れなく含めるだけで、AIの出力品質が安定します。

| ステップ | 要素 | やること | 例 |
|---------|------|---------|-----|
| 1 | 役割（Role） | AIの立場を定義する | 「あなたは中小企業の経理担当です」 |
| 2 | タスク（Task） | 具体的な処理内容を指示する | 「問い合わせメールをカテゴリに分類してください」 |
| 3 | 出力（Output） | 出力の形式・構造を指定する | 「JSON形式で返してください」 |
| 4 | 制約（Constraints） | 条件・制限・注意点を追加する | 「50文字以内で要約。不明な場合は"不明"と出力」 |

### なぜ構造化が必要なのか

曖昧なプロンプトとフレームワークに沿ったプロンプトでは、出力の質に大きな差が出ます。

| | 曖昧なプロンプト | 構造化されたプロンプト |
|---|----------------|-------------------|
| 入力例 | 「このメールを分析して」 | 「あなたは問い合わせ対応の担当者です。以下のメールをカテゴリ（見積依頼/質問/クレーム）に分類し、JSON形式で出力してください」 |
| AIの出力 | 自由形式の長文分析（毎回フォーマットが変わる） | 指定どおりのJSON（プログラムで処理しやすい） |
| 業務への組み込み | 手動で読んで判断が必要 | そのまま自動処理に接続できる |

業務自動化では、AIの出力をプログラムで処理することが前提です。出力が毎回異なるフォーマットでは自動化が成立しません。構造化されたプロンプトが不可欠な理由はここにあります。

## 4ステップの実践

4ステップの実践とは、フレームワークの各要素を順番に組み立てて実用的なプロンプトを完成させる作業のことです。

### ステップ1: 役割を定義する

AIに業務上の役割を与えることで、出力のトーンや専門性が適切になります。

```text
あなたは中小企業の経理担当です。
```

**効果的な役割定義のコツ:**

| コツ | 悪い例 | 良い例 |
|------|--------|--------|
| 業務領域を明確にする | 「あなたはAIです」 | 「あなたは中小企業の経理担当です」 |
| 専門性を指定する | 「回答してください」 | 「10年の経験を持つ経理担当として回答してください」 |
| 対象読者を伝える | （なし） | 「事務担当者が読むことを想定して平易に説明してください」 |

### ステップ2: タスクを具体的に指示する

「何を」「どうするか」を具体的に書きます。曖昧さが残ると、AIが意図と異なる処理をします。

```text
以下の問い合わせメール本文を分析し、カテゴリを判定してください。
カテゴリ: 見積依頼 / 製品問い合わせ / クレーム / その他
```

**具体化のポイント:**

- 動詞を明確にする: 「分析して」ではなく「分類して」「要約して」「抽出して」
- 選択肢を限定する: 「適切なカテゴリに分類」ではなく具体的な選択肢を列挙
- 処理単位を明示する: 「メール1通ずつ」「行ごとに」「全体をまとめて」

### ステップ3: 出力形式を指定する

業務自動化で最も重要なステップです。出力形式が安定していないと後続の処理に接続できません。

```text
以下のJSON形式のみで返してください：
{"category": "カテゴリ名", "summary": "50文字以内の要約", "urgency": "高/中/低"}
```

**業務自動化に適した出力形式:**

| 出力形式 | 用途 | 後続処理の例 |
|---------|------|------------|
| JSON | API連携・プログラム処理 | データベースへの保存、条件分岐 |
| CSV / TSV | スプレッドシート連携 | GASでスプレッドシートに転記 |
| 箇条書き | レポート・通知 | Slackへの投稿、メール本文 |
| 表形式（Markdown） | レビュー・比較 | ドキュメント生成 |

[Claude API入門](/articles/ai-api/claude-api-intro)や[OpenAI API入門](/articles/ai-api/openai-api-intro)でJSON出力を活用した実装例を掲載しています。

### ステップ4: 制約・条件を追加する

出力の品質を安定させるために、制約条件を明記します。

```text
- 1つのカテゴリのみ選択すること（複数選択不可）
- 判断できない場合は category を "その他" とすること
- summary は必ず50文字以内に収めること
- JSON以外のテキストは一切出力しないこと
```

**よく使う制約パターン:**

| 制約の種類 | 具体例 | 効果 |
|-----------|--------|------|
| 文字数制限 | 「50文字以内で」 | 出力が冗長になるのを防ぐ |
| 出力限定 | 「JSON以外のテキストを出力しない」 | パース（構文解析）エラーを防ぐ |
| 不明時の処理 | 「判断できない場合は"不明"と出力」 | エラーハンドリングが容易に |
| トーンの指定 | 「です・ます調で」 | 文体の統一 |
| 禁止事項 | 「推測で情報を補完しない」 | 虚偽情報の混入を防ぐ |

### 完成形：4ステップを組み合わせたプロンプト

上記の4ステップを組み合わせると、以下のようなプロンプトが完成します。

```text
あなたは中小企業の問い合わせ対応担当です。

以下の問い合わせメール本文を分析し、カテゴリを判定してください。
カテゴリ: 見積依頼 / 製品問い合わせ / クレーム / その他

以下のJSON形式のみで返してください：
{"category": "カテゴリ名", "summary": "50文字以内の要約", "urgency": "高/中/低"}

制約:
- 1つのカテゴリのみ選択すること
- 判断できない場合は category を "その他" とすること
- JSON以外のテキストは一切出力しないこと
```

このプロンプトは[Claude API入門](/articles/ai-api/claude-api-intro)のメール分類スクリプトで実際に使用しているものです。systemプロンプトに設定し、メール本文をuserメッセージとして送信する構成で動作します。

## 業務シーン別テンプレート集

業務シーン別テンプレート集とは、4ステップのフレームワークを中小企業の代表的な業務に適用した、すぐに使えるプロンプト例のことです。

### テンプレート1: レポート・議事録の要約

```text
[役割] あなたは業務レポートの編集担当です。
[タスク] 以下のテキストを要約してください。
[出力] 以下の形式で出力してください：
  - 要点（3〜5個の箇条書き）
  - 次のアクション（あれば箇条書き）
  - 一言まとめ（30文字以内）
[制約]
  - 原文にない情報を追加しない
  - 固有名詞・数値はそのまま残す
  - です・ます調で統一
```

### テンプレート2: 顧客データの整形・分類

```text
[役割] あなたはデータ整理の担当者です。
[タスク] 以下のCSVデータを読み込み、指定の形式に変換してください。
[出力] CSV形式で出力してください。ヘッダー行を含めること。
  出力カラム: 会社名, 担当者名, メールアドレス, 都道府県, 業種
[制約]
  - 都道府県は住所から抽出すること
  - 業種が不明な場合は "未分類" とすること
  - 元データの行数と出力の行数が一致すること
```

### テンプレート3: 定型メールの下書き生成

```text
[役割] あなたは営業事務の担当者です。
[タスク] 以下の情報をもとに、お礼メールの下書きを作成してください。
  - 宛先: {顧客名} 様
  - 用件: {打ち合わせ/ご注文/お問い合わせ} のお礼
  - 補足: {補足情報}
[出力] 件名と本文を以下の形式で出力してください：
  件名: ...
  本文: ...
[制約]
  - です・ます調で統一
  - 200文字以内に収めること
  - 社名は「弊社」と表記
```

これらのテンプレートは[Claude Code](/articles/ai-api/claude-code-automation)に直接貼り付けて使うことも、[Claude API](/articles/ai-api/claude-api-intro)のsystemプロンプトに設定してプログラムから呼び出すことも可能です。

## よくある失敗と改善のコツ

よくある失敗と改善のコツとは、プロンプト設計で初心者が陥りやすい問題とその対処法をまとめたチェックリストのことです。

| 失敗パターン | 原因 | 改善策 |
|------------|------|--------|
| 毎回出力のフォーマットが違う | 出力形式の指定が曖昧 | JSON等の構造を具体的に例示する |
| AIが余計な前置きを出力する | 制約が不足 | 「指定の形式以外のテキストを出力しない」を追加 |
| 要約が長すぎる | 文字数制限が未指定 | 「〇文字以内」と明示する |
| 分類結果が安定しない | 選択肢が曖昧 | 選択肢を列挙し、各カテゴリの判断基準を追記 |
| 実在しない情報を生成する | ハルシネーション | 「入力にない情報を補完しない」「不明な場合は"不明"と出力」を追加 |

### プロンプト改善チェックリスト

プロンプトがうまく動かないときは、以下を確認してください。

- [ ] 役割は業務の文脈に合っているか
- [ ] タスクに曖昧な動詞（「分析して」「処理して」）を使っていないか
- [ ] 出力形式は具体的に例示されているか
- [ ] 不明・例外時の処理ルールを指定しているか
- [ ] 不要な出力を禁止する制約を入れているか

### temperatureパラメータの使い分け

APIでプロンプトを使う場合、temperatureパラメータ（出力のランダム性を制御する値）の設定も重要です。

| 用途 | temperature | 理由 |
|------|------------|------|
| メール分類・データ整形 | 0〜0.3 | 安定性重視。毎回同じ結果が望ましい |
| 要約・レポート生成 | 0.3〜0.7 | 適度な表現の幅が必要 |
| メール下書き・文章生成 | 0.7〜1.0 | 多様な表現を生成したい |

APIのtemperature設定方法は[Claude API入門](/articles/ai-api/claude-api-intro)や[OpenAI API入門](/articles/ai-api/openai-api-intro)で解説しています。
